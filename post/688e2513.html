消息队列之Kafka

Posted on 2025-07-26 Word count in article: 4.7k Reading time ≈ 17 mins.

Kafka

  • Kafka/RocketMQ/RabbitMQ: 深入理解其核心概念（Topic/Queue, Producer, Consumer, Broker, Partition/Sharding）、消息存储机制（Kafka 的 Log Segment, RocketMQ 的 Commit Log）、高可用与可靠性保证（副本机制、ACK 机制、事务消息）、顺序消息、延迟消息、消息积压处理策略。理解选型考量。

基本概念

  • Topic 是消息发布的类别或主题。
  • Partition：一个 Topic 被划分为一个或多个 Partition，分布在不同的 Broker 上。
  • Replica：每个 Partition 可以有多个 Replica (副本)。其中一个 Replica 是 Leader，负责读写；其他是 Follower，只负责从 Leader 同步数据。Replica 也分布在不同的 Broker 上，提供容错能力。
  • Consumer Group (CG): 一组 Consumer 实例的集合，它们共同消费一个或多个 Topic。组内消费实例协作消费订阅 Topic 的所有 Partition。一个 Partition 只能被组内一个 Consumer 实例消费，实现负载均衡。
  • Rebalance (重平衡): 当以下情况发生时，Kafka 会重新分配 CG 内 Consumer 负责的 Partition：
    • 组内 Consumer 数量变化 (新 Consumer 加入、Consumer 崩溃或主动离开)。
    • 订阅的 Topic Partition 数量变化 (如 Topic 被修改增加了 Partition)。
    • 订阅的 Topic 本身变化。
    • 影响: Rebalance 期间，整个 CG 会短暂停止消费，直到分配完成。频繁或长时间的 Rebalance 会影响消费进度和系统稳定性。

Kafka 如何实现高吞吐量

  • 顺序读写磁盘: 消息按顺序追加到 Partition 文件末尾，充分利用磁盘顺序 I/O 的高性能（远高于随机 I/O）。
  • Page Cache: 利用操作系统 Page Cache 缓存数据，减少直接磁盘访问。
  • 零拷贝 (Zero-Copy): 使用 sendfile 和 mmap 等技术，减少数据在用户空间和内核空间之间的拷贝次数，降低 CPU 开销和上下文切换。
  • 批量处理 (Batching): Producer 批量发送消息，Broker 批量写入磁盘和批量发送给 Consumer，减少网络和 I/O 开销。
  • 消息压缩: Producer 端可配置压缩算法 (Snappy, LZ4, Gzip, Zstd)，减少网络传输和磁盘存储开销。
  • Partition 并行: Topic 被分成多个 Partition，分布在多个 Broker 上，读写操作可以并行发生在不同 Partition，充分利用多 Broker、多磁盘、多核 CPU。

Producer 的 acks 不同取值的含义

  • acks=0: Producer 发送消息后不等待 Broker 的任何确认。最高吞吐，最低延迟，最低可靠性 (可能丢失数据)。
  • acks=1: Producer 发送消息后，等待 Leader 副本成功写入本地日志 的确认。折中方案。Leader 故障但数据未同步到 Follower 时可能丢失数据。
  • acks=all (或 acks=-1): Producer 发送消息后，等待 Leader 收到消息并确认该消息已被当前 ISR 集合中的所有副本成功写入本地日志。最高可靠性，最低丢失风险，但延迟最高，吞吐可能最低。需要配合 min.insync.replicas 参数使用。

如何保证消息不丢失

消息丢失可能发生在生产者、Broker 或消费者环节：

  1. 生产者发送消息时丢失：
    • 原因：
      • 异步发送 + 不处理回调： 生产者使用异步发送（send() 返回 Future）但未正确处理回调（Future.get() 或 Callback），网络抖动或 Broker 暂时不可用导致发送失败，但生产者不知情。
      • acks 配置过低： acks=0（生产者不等待任何确认）或 acks=1（只等待 Leader 写入本地日志）时，如果 Leader 刚写入就挂掉且数据未同步到 Follower，新 Leader 可能没有这条消息。
    • 解决方案：
      • 使用同步发送或正确处理异步回调： 调用 future.get() 阻塞等待结果，或在 Callback 中检查异常并实现重试逻辑（注意幂等性）。
      • 设置 acks=all： 要求 Leader 收到消息后，必须等待所有 ISR (In-Sync Replica) 列表中的 Follower 都成功复制了消息后才向生产者发送确认。这是防止生产者端丢失的最强保证。
      • 配置合理的 retries 和 retry.backoff.ms： 对于可重试的异常（如网络问题、NOT_ENOUGH_REPLICAS），生产者应自动重试。增大重试次数和重试间隔。
      • 启用生产者幂等性： 设置 enable.idempotence=true。这会自动设置 acks=all，并引入生产者 ID 和序列号，防止在重试时因网络问题导致 Broker 收到重复消息（解决了重试可能导致重复的问题，但主要目标是保证精确一次生产语义的基础）。
  2. Broker 存储消息时丢失：
    • 原因：
      • 副本数量不足且 Leader 挂掉： 假设 replication.factor=1（只有 Leader 副本），如果该 Broker 磁盘损坏或永久宕机，消息永久丢失。即使 replication.factor>1，但如果 min.insync.replicas=1 且 acks=all，只要有一个副本（Leader）写入成功就算成功，如果该 Leader 立刻挂掉且数据未同步到其他副本，消息也会丢失。
      • 不干净的 Leader 选举： 当所有 ISR 副本都不可用时（例如全挂了），如果 unclean.leader.election.enable=true，Kafka 会从非 ISR（可能落后很多）的副本中选举新 Leader。这个新 Leader 会丢失掉它未复制到的所有消息。
    • 解决方案：
      • 设置足够的 replication.factor： 至少设置为 3。这是生产环境的标准配置，确保即使一个 Broker 永久故障，数据也不丢失。
      • 设置 min.insync.replicas： 通常设置为 replication.factor - 1（例如 RF=3 时 min.isr=2）。这定义了当生产者使用 acks=all 时，Leader 必须等待多少个 ISR 副本成功写入后才返回确认。结合 acks=all 和 min.insync.replicas，可以确保写入成功的消息至少存在于 min.insync.replicas 个副本上，大大提高了数据持久性。
      • 设置 unclean.leader.election.enable=false： 强烈建议生产环境关闭！ 这阻止了从非 ISR 副本选举 Leader，宁可让分区不可用（影响可用性），也要保证数据一致性（不丢失已确认的消息）。需要通过监控和运维手段快速恢复 ISR 副本以保证可用性。
  3. 消费者拉取消息时丢失：
    • 原因：
      • 自动提交偏移量 + 消息处理失败： 消费者默认或在处理消息前自动提交偏移量。如果消息在处理过程中（例如业务逻辑出错、消费者崩溃）失败，但偏移量已经提交，那么这条消息对于该消费者组来说就“丢失”了（因为下次拉取会从已提交的偏移量之后开始）。虽然消息还在 Broker 上，但没有消费者会再处理它。
    • 解决方案：
      • 关闭自动提交，采用手动提交偏移量： 设置 enable.auto.commit=false。
      • 在处理完消息后手动提交偏移量： 在业务逻辑成功执行完毕后，再调用 consumer.commitSync() 或 consumer.commitAsync() 提交偏移量。这保证了“至少一次”语义（消息可能被处理多次，但至少处理一次）。
      • 注意提交的原子性： 尽量保证处理单条消息后立即提交其偏移量。批量处理时，需要在成功处理完一个批次后再提交该批次中最后一条消息的偏移量。如果处理失败，不要提交偏移量，这样消费者重启后会重新拉取失败的消息。
      • 处理好再平衡： 在消费者发生再平衡（Consumer Rebalance - 消费者加入或离开组）时，确保在失去分区所有权前提交已处理消息的偏移量。监听 ConsumerRebalanceListener 接口，在 onPartitionsRevoked 方法中进行提交。

如何解决消息重复消费的问题

消息重复消费主要发生在消费者端：

  1. 原因：
    • 消费者处理成功但提交偏移量失败： 这是最常见的原因。消费者成功处理了消息，但在提交偏移量时失败（例如网络问题、消费者崩溃、再平衡）。当消费者恢复或新消费者接管分区后，会从上一次成功提交的偏移量开始消费，导致已处理但未提交偏移量的消息被再次处理。
    • 生产者重试导致消息重复写入： 在未启用幂等性时，生产者发送消息后未收到 Broker 确认（可能 Broker 已写入但确认丢失），生产者重试可能导致 Broker 收到两条相同的消息（具有不同的偏移量）。
    • 消费者再平衡： 再平衡过程中，如果消费者在提交偏移量前被撤销分区，且新消费者在旧消费者完成处理前接管，可能导致消息被不同消费者重复处理（尤其是处理时间较长时）。
    • 消费者 Seek： 手动将消费者偏移量重置到之前的位置会导致重复消费。
  2. 解决方案：
    • 启用生产者幂等性： 设置 enable.idempotence=true。这是解决生产者重试导致 Broker 端消息重复的根本方法。它确保单个生产者会话内发送到同一分区的消息是幂等的。
    • 启用 Kafka 的事务： 对于需要跨 Kafka Topic 和外部系统（如数据库）保证“精确一次”语义的场景，需要使用 Kafka 事务 API。生产者使用事务发送消息，消费者在事务内读取消息并提交偏移量（设置 isolation.level=read_committed）。这需要外部系统也支持事务性参与（通常通过两阶段提交或事务性出站）。
    • 消费者端实现幂等性： 这是最常用、最灵活、适用范围最广的解决方案！ 既然 Kafka 消费者端默认是“至少一次”语义，那么就需要你的消费逻辑能够容忍重复。核心思想是让处理操作本身具有幂等性：
      • 唯一标识： 在消息体中携带一个全局唯一的业务标识符（如订单ID、支付流水号）。
      • 状态检查： 在处理消息前，先根据这个唯一标识去检查目标系统（通常是数据库）中是否已经存在相同标识的处理结果。
      • 幂等操作： 如果存在，则跳过处理或执行更新操作（确保更新操作本身也是幂等的，例如 UPDATE table SET status='paid' WHERE order_id=123 AND status='unpaid'）。如果不存在，则正常处理。
      • 数据库唯一约束： 在数据库表中，利用唯一键约束来防止重复插入是最有效的手段。尝试插入带有唯一标识的记录，如果违反唯一约束则视为重复消息，进行更新或忽略。

Kafka 如何保证消息的顺序性

  • Kafka 仅保证单个 Partition 内消息的严格顺序 (FIFO)。 这是由其追加写入和单线程消费分区的特性保证的。
  • 无法保证跨 Partition 的消息全局顺序。
  • 需要全局顺序怎么办？ 将所有需要保证顺序的消息发送到同一个 Partition。通常通过为这些消息指定相同的 Key 来实现 (Producer 根据 Key 的哈希值决定 Partition)。
  • 需要分区顺序怎么办？ 确保同一个 Key 的消息总是发到同一个 Partition (同样通过 Key 哈希)，则该 Key 相关的消息在该 Partition 内是顺序的。

ISR 机制及其在 Leader 选举中的作用

ISR (In-Sync Replicas): 指与 Partition Leader 副本保持同步的 Follower 副本集合。Leader 维护 ISR 列表。

同步标准: Follower 副本在 replica.lag.time.max.ms 时间内成功从 Leader 拉取到最新数据。

Leader 选举: 当 Leader 副本所在 Broker 宕机时，Controller 会从当前 ISR 集合中选择一个副本作为新的 Leader。这是为了保证新 Leader 拥有最新的已提交数据，避免数据丢失或不一致。

Zookeeper 和 KRaft 的区别，为什么 Kafka 要去 Zookeeper 化

  • Zookeeper (ZK)： 是 Kafka 2.7 及之前版本的核心外部依赖。负责：
    • Broker 注册与发现
    • Controller 选举
    • Topic 配置存储
    • ACL 权限存储
    • (旧版) Consumer Offset 存储
  • KRaft： 是 Kafka 内部实现的基于 Raft 共识协议的元数据管理系统，从 Kafka 2.8 (预览) / 3.0+ (生产可用) 开始引入，旨在完全取代 Zookeeper。
  • 去 ZK 化原因 (KRaft 优势)：
    • 简化架构与部署： 移除外部依赖，Kafka 自包含，部署、配置、监控更简单。
    • 提升性能与可伸缩性： 元数据操作直接在 Kafka 协议上处理，减少一次网络跳转和序列化开销。KRaft 设计上能更好地支持大规模集群（数万个 Partition）。
    • 更强的元数据一致性模型： KRaft 使用 Raft 强一致性协议管理元数据状态机。
    • 更直接的运维： 所有操作通过 Kafka API 或 kafka-metadata-shell 完成，无需单独管理 ZK。
    • 统一安全模型： 元数据通信复用 Kafka 的安全机制 (SSL/SASL)，无需单独配置 ZK 安全。
    • 现状： KRaft 已成为生产推荐模式，新集群应优先使用 KRaft。Zookeeper 模式在逐步淘汰。

什么是 Consumer Lag

  • 定义： Consumer Lag 指 Consumer Group 在某个 Partition 上最新提交的 Offset 与 Partition 当前最新消息的 Offset (LEO) 之间的差值。即 未消费的消息数量。
  • 监控重要性： Lag 是衡量消费健康状况的关键指标。持续增长的 Lag 意味着消费速度跟不上生产速度，可能导致延迟和数据积压
  • 原因：
    • 消费者处理能力不足 (CPU, 内存, 逻辑复杂, I/O 慢)。
    • 消费者实例数量不足 (Partition 数 > Consumer 实例数)。
    • 下游系统故障或慢。
    • 消费者代码 Bug 或配置问题 (如 max.poll.records 太大导致单次处理时间过长)。
    • 网络问题。
  • 处理：
    • 监控: 使用 Kafka 自带工具 (kafka-consumer-groups.sh)，JMX 指标，或第三方监控系统 (Prometheus+Grafana, Confluent Control Center)。
    • 扩容： 增加 Consumer Group 内的 Consumer 实例数 (确保 <= Partition 数)。
    • 优化消费者： 提升单 Consumer 处理能力 (优化代码逻辑、异步处理、批处理、调优 JVM)。
    • 调整参数： 适当增加 fetch.min.bytes, fetch.max.wait.ms 提高拉取效率；调整 max.poll.records 避免单次处理过多消息超时；检查 session.timeout.ms 和 max.poll.interval.ms 避免误判离线触发 Rebalance。
    • 排查下游依赖。

什么是死信队列

死信队列（Dead Letter Queue，简称 DLQ）是消息队列系统（如 RabbitMQ、Kafka、ActiveMQ、AWS SQS 等）中的一个核心容错机制。它的核心作用在于处理那些无法被正常消费或传递的消息，你可以把它想象成一个专门收容“失败任务”的隔离区。

简单来说，当一个消息在系统中“死掉”（无法被成功处理）时，它就会被转移到这个特殊的队列（DLQ）中，而不是被直接丢弃或无限次重试导致系统阻塞。

消息为什么会“死”（进入死信队列）？

通常有以下几种常见原因：

  1. 消息被拒绝且未重新入队：
    • 消费者在处理消息时遇到无法处理的错误（如业务逻辑错误、数据格式不合法、依赖服务不可用等）。
    • 消费者显式地发送了否定确认（NACK）并且设置了 requeue=false（或者在某些系统中是显式拒绝），表示此消息不应再放回原队列尝试消费。
  2. 消息过期（TTL 过期）：
    • 消息在队列中等待的时间超过了预设的生存时间（Time-To-Live， TTL）。
    • 消息在等待投递给消费者时超过了其预设的 TTL（某些系统支持）。
  3. 队列达到最大长度限制：
    • 当目标队列已满（达到了配置的最大消息数量限制），新到达的消息无法入队时，这些消息可能会被转移到 DLQ（取决于配置）。
  4. 投递失败次数超限：
    • 消息被消费者多次尝试消费（例如，配置了重试机制），但均告失败，达到了最大重试次数上限。此时系统会认为该消息无法被成功处理，将其放入 DLQ。
